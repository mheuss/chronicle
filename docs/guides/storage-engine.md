# Storage Engine

Developer guide for the `chronicle-storage` crate.

## What it does

This crate owns all persistent data. It stores screenshot metadata and audio
segment metadata in SQLite, keeps FTS5 full-text search indexes in sync with
triggers, manages on-disk media files in a date-bucketed directory tree, and
handles retention-based cleanup.

The database file lives at `<base_dir>/chronicle.db`. Media files live under
`<base_dir>/screenshots/` and `<base_dir>/audio/`. On macOS the default base
directory is `~/Library/Application Support/Chronicle`.

WAL mode is enabled on every connection so the UI can search while the daemon
writes. See [ADR-007](../decisions/007-wal-mode-sqlite.md) for why.

## Data flow

### Normal capture flow

1. **Allocate path.** The caller asks for a file path via
   `allocate_screenshot_path()` or `allocate_audio_path()`. The crate builds a
   date-bucketed path (`screenshots/2026/03/21/1774094400000_display1.heif`) and
   creates the parent directories.
2. **Caller writes the file.** The capture or audio crate writes the actual
   image or audio data to the path it received.
3. **Insert metadata.** The caller calls `insert_screenshot()` or
   `insert_audio_segment()` with a metadata struct. This runs an INSERT into the
   main table.
4. **FTS triggers fire.** The INSERT triggers (`screenshots_ai`, `audio_ai`)
   automatically copy text columns into the corresponding FTS5 table. The index
   is searchable immediately.

### Late-update flow (OCR / transcription)

Screenshots are often inserted before OCR finishes. Audio segments are inserted
before transcription completes. The crate handles this with update methods:

1. `update_ocr_text(id, text)` runs an UPDATE on the `screenshots` table.
2. The `screenshots_au` trigger fires, deletes the old FTS entry, and inserts
   the new one.
3. The same pattern applies for `update_transcript(id, text)` on
   `audio_segments`.

This means a row can be inserted with `ocr_text = NULL`, and the FTS index
updates correctly once the real text arrives.

## Module structure

```text
chronicle-daemon/crates/storage/src/
├── lib.rs             Storage struct, async wrappers, connection pool setup
├── error.rs           StorageError enum (Database, Pool, Io, Join, Other)
├── models.rs          All public data types (configs, metadata, results, stats)
├── schema.rs          Connection PRAGMAs, migration runner (user_version-based)
├── files.rs           Date-bucketed path generation, directory creation
├── screenshots.rs     Insert, get, get_timeline, update_ocr_text
├── audio.rs           Insert, get, update_transcript
├── search.rs          Unified FTS5 search across screenshots and audio
├── retention.rs       Batch cleanup, orphan file sweep
└── migrations/
    └── 001_initial_schema.sql
```

`lib.rs` is the public API surface. Every public method is `async` and uses
`tokio::task::spawn_blocking` to run synchronous rusqlite calls on the blocking
thread pool. The inner modules (`screenshots`, `audio`, `search`, `retention`,
`files`, `schema`) are all `pub(crate)` — callers only interact through the
`Storage` struct.

Connection pooling uses `r2d2`. A `ConnectionCustomizer` runs
`setup_connection()` on every new connection to set PRAGMAs: `journal_mode=WAL`,
`synchronous=NORMAL`, `foreign_keys=ON`, `busy_timeout=5000`.

## How search works

Search uses external content FTS5 tables. See
[ADR-006](../decisions/006-external-content-fts5.md) for the design rationale.

**FTS5 tables:**

- `screenshots_fts` indexes `ocr_text`, `app_name`, and `window_title` from the
  `screenshots` table.
- `audio_fts` indexes `transcript` from the `audio_segments` table.

Both use `content=<table>, content_rowid=id` so there is no data duplication.

**Triggers keep it in sync.** Each main table has three triggers (after insert,
after update, after delete) that mirror changes into the FTS table. The update
trigger does a delete-then-insert to replace the old entry.

**Query path (`search::search`):**

1. If the filter includes screenshots, query `screenshots_fts` with `MATCH`,
   JOIN back to `screenshots` for full row data, use `snippet()` for context,
   and collect results with `rank`.
2. If the filter includes audio, do the same against `audio_fts` /
   `audio_segments`.
3. Merge both result sets, sort by `rank` (lower is better in FTS5), then apply
   `limit` and `offset`.

**Filters:** `SearchFilter::All` queries both tables. `ScreenOnly` and
`AudioOnly` query one.

**Snippets:** Generated by `snippet(fts_table, -1, '<b>', '</b>', '...', 32)`.
The caller gets an HTML-ish string with bold markers around matched terms and
ellipsis for context trimming.

## How to add a new column

Say you want to add `browser_url` to screenshots.

1. **Create a new migration file.** Add
   `migrations/002_add_browser_url.sql` with the `ALTER TABLE` statement. Do not
   modify `001_initial_schema.sql` — existing databases have already run it.
2. **Register the migration.** In `schema.rs`, add the new file to the
   `MIGRATIONS` array: `include_str!("migrations/002_add_browser_url.sql")`.
   The runner uses `PRAGMA user_version` to track which migrations have run. It
   will run the new one automatically on next open.
3. **Update the model structs.** Add the field to `ScreenshotMetadata` and
   `Screenshot` in `models.rs`.
4. **Update queries.** Add the column to the INSERT in `screenshots::insert`,
   the SELECT in `screenshots::get` and `screenshots::get_timeline`, and the
   `row_to_screenshot` mapper.
5. **If the column is searchable,** update the FTS5 table definition and its
   triggers. This likely means a second migration that recreates the FTS table
   and triggers (FTS5 virtual tables don't support `ALTER TABLE`).
6. **Add tests.** At minimum, test that insert round-trips the new field and
   that search picks it up (if it's indexed).

## How cleanup works

Retention cleanup runs in two phases inside `retention.rs`.

### Phase 1: Expire old rows (`run_cleanup`)

1. Read `retention_days` from the `config` table (default 30).
2. Compute the cutoff timestamp: `now - retention_days * 86400 * 1000`.
3. For screenshots: select up to 500 rows older than cutoff, DELETE them, then
   delete each associated file from disk. Repeat until no more expired rows.
4. Same loop for audio segments.

Batching at 500 rows keeps SQLite transactions short. The DELETE triggers
automatically clean up FTS entries. DB rows are deleted first so that if the
file delete fails, the row is still gone and the file becomes an orphan (caught
by phase 2).

### Phase 2: Sweep orphan files (`sweep_orphans`)

Walks `screenshots/` and `audio/` directories recursively. For each file, checks
if a row in the corresponding table references that path. If not, deletes the
file. This catches files left behind by crashes or interrupted cleanup.

## How to run tests

Unit tests use in-memory SQLite (no disk I/O, no cleanup needed). File-related
tests use `tempfile::tempdir()` which auto-cleans on drop.

```sh
# Run all storage tests
cargo test -p chronicle-storage

# Run a specific test
cargo test -p chronicle-storage -- search_finds_both_with_shared_term

# Run the integration test (full async workflow)
cargo test -p chronicle-storage --test integration
```

The integration test in `tests/integration.rs` exercises the full async API
end-to-end: path allocation, file writes, metadata inserts, OCR/transcript
updates, search, timeline queries, config, status, and cleanup.
